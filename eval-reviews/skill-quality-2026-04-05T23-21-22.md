### Root Causes

1. **Finding**: Agent never ran `kata can-exit`
   **Broke at**: Stage 5 — agent
   **Why**: After marking task #2 complete, the agent switched into "report mode" — producing a summary table of skill activations rather than continuing the pipeline. The agent treated task completion as session completion.
   **Fix**: Template should include explicit exit instructions (e.g., a final "Exit" section saying "Run `kata can-exit` and address any unmet conditions"). Alternatively, the Stop hook should fire when the agent produces a final-sounding message without having run `can-exit`.

2. **Finding**: Changes never committed despite template instruction and global condition
   **Broke at**: Stage 4 — agent
   **Why**: P1 TDD instruction lists "4. Commit your changes" as the final sub-step, but the agent stopped after verifying the build. The agent may have treated the commit instruction as optional or post-session cleanup. The `global_conditions: changes_committed` should have been a hard gate but was never checked because `can-exit` was never called.
   **Fix**: Make the commit step a separate task (not buried as sub-step 4 of a multi-part instruction) OR add a dependency hook that blocks task #2 completion until changes are committed.

3. **Finding**: No `TaskList` call after mode entry
   **Broke at**: Stage 2 — agent
   **Why**: The agent received the phase list in the `kata enter` JSON response (`"phases": ["p0", "p1"]`) and inferred task IDs from phase ordering. This happened to work but bypasses the formal task discovery stage.
   **Fix**: Enter output should explicitly instruct "Call TaskList to see your pre-created tasks and their dependencies" rather than just listing phase IDs.

4. **Finding**: P0 plan only covered 2 of 5 methodology steps
   **Broke at**: Stage 3 — agent
   **Why**: The agent did extensive exploration (read specs, routes, config files — ~20+ tool calls) but when writing the plan output, only produced "Understand the Request" and "Scope the Work" sections before marking P0 complete. Possible the plan was longer but truncated in the transcript, or the agent considered the exploration itself as satisfying steps 3–5.
   **Fix**: Template should define what "done" looks like for P0 (e.g., "Plan must include all 5 sections: understanding, scope, risks, verification, steps"). The quick-planning skill's SKILL.md should enforce this as a checklist.

5. **Finding**: Repeated `find -type f` failures causing sibling cascades
   **Broke at**: Stage 3 — agent (minor)
   **Why**: The sandbox environment aliases `find` in a way that transforms `-type` to `-ype`. Agent used `find` via Bash instead of using `Glob` tool. After first failure, agent did adapt to `Glob`, but retried `find` once more before switching permanently.
   **Fix**: Agent training issue — should prefer `Glob` over `find` per tool documentation.
